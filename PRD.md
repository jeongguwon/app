# 스쿨줍줍 PRD (Product Requirements Document)

기획([IDEATION.md](IDEATION.md))과 기술 아키텍처([ARCHITECTURE.md](ARCHITECTURE.md))를 통합한 개발 실행 문서. MVP 출시까지의 전체 범위·태스크·인수 조건·일정·릴리즈 기준을 정의한다.

---

## 1. 제품 요약

| 항목 | 내용 |
|---|---|
| 제품명 | 스쿨줍줍 |
| 한 줄 소개 | 학교 구성원 전용 분실물·습득물 매칭 PWA |
| 대상 사용자 | 단일 학교의 학생·교직원(이메일 인증된 사람만) |
| 플랫폼 | 모바일 우선 반응형 웹앱(PWA) |
| 출시 목표 | MVP 1회 출시 후 학교 1곳 파일럿 |

## 2. 제품 목표와 비목표

### 2.1 목표 (In Scope)
- 학교 인스타그램 대비 분실물 매칭 소요 시간을 절반 이하로 단축
- 인증된 학교 구성원만 사용 가능한 폐쇄형 서비스
- 비공개 식별 질문 기반의 안전한 소유권 검증
- 전화번호 노출 없는 연락 흐름
- 자동 만료·삭제로 개인정보 최소 보관

### 2.2 비목표 (Out of Scope, v2 이후)
- 인앱 1:1 실시간 채팅
- 이미지 유사도 자동 매칭
- 다중 학교 동시 운영
- 네이티브 앱 출시
- 다국어(영어 등) 지원
- 결제·포인트 시스템

## 3. 성공 지표 (KPI)

| 지표 | 목표값(파일럿 3개월) | 측정 방법 |
|---|---|---|
| 평균 매칭 소요 시간 | < 48시간 | `returned_at - created_at` 평균 |
| 분실물 회수율 | > 40% | 반환 완료 / 전체 등록 |
| MAU / 가입자 | > 30% | 30일 내 로그인 사용자 비율 |
| 학교 공식 계정 분실물 게시 감소 | > 50% | 운영팀 수동 집계 |
| 신고 발생률 | < 5% | report 건수 / post 건수 |

## 4. 사용자 페르소나

| 페르소나 | 시나리오 | 핵심 니즈 |
|---|---|---|
| 잃어버린 학생 | 강의실에 에어팟을 두고 나옴, 다음날 발견 | 누가 주웠는지 빠르게 확인하고 회수 |
| 주운 교직원 | 도서관 데스크에서 지갑 발견 | 주인 찾아 돌려주되 본인 정보는 보호 |
| 학생회 관리자 | 신고 들어온 부적절 게시물 처리 | 빠른 모니터링·조치 도구 |

## 5. 핵심 사용자 흐름 (User Journey)

### 5.1 분실자 흐름
```
가입(이메일 OTP) → 홈 검색 → 습득물 글 발견 →
"내 물건 같아요" → 비공개 질문 답변 → 통과 →
연락 채널 오픈 → 만남·인계 → "수령 확인"
```

### 5.2 습득자 흐름
```
가입 → 습득물 등록(사진·카테고리·장소·비공개 질문) →
주인 클레임 도착 → 답변 확인·수동 승인 →
연락 채널 오픈 → 인계 → "수령 확인"
```

### 5.3 관리자 흐름
```
관리자 콘솔 → 신고 큐 → 게시물 확인 → 숨김/삭제 →
신고자·작성자 통지
```

## 6. 기능 요구사항 (Functional Requirements)

각 항목은 **요구사항 ID** · **설명** · **인수 조건(Acceptance Criteria)** 으로 구성.

### FR-1. 회원가입 / 로그인 (이메일 OTP)

**설명**: 학교 도메인 화이트리스트 기반 이메일 OTP 인증.

**인수 조건**
- [ ] 화이트리스트에 등록된 도메인(@school.ac.kr 등)만 OTP 발송 가능
- [ ] 6자리 코드 발송, 5분 유효, 최대 3회 재전송
- [ ] 로그인 실패 5회 시 15분 잠금
- [ ] 가입 시 닉네임 입력(2~12자, 중복 불가)
- [ ] 14세 미만 가입 시 법정대리인 동의 화면 표시
- [ ] 로그아웃 / 회원 탈퇴 가능 (탈퇴 시 즉시 데이터 익명화)

### FR-2. 게시글 등록 (분실물 / 습득물)

**설명**: 사진·메타데이터를 함께 등록하며 EXIF는 자동 제거.

**인수 조건**
- [ ] 사진 1~4장, 각 5MB 이하, jpg/png/webp 만 허용
- [ ] 클라이언트에서 최대 1600px 리사이즈, EXIF/GPS 제거 후 업로드
- [ ] 서버에서 sharp로 EXIF 한 번 더 제거 (이중 방어)
- [ ] 필수 입력: 종류, 카테고리, 제목, 장소, 일시
- [ ] 습득물: 보관 장소 + 비공개 식별 질문 필수
- [ ] 분실물: 비공개 식별 정보 입력 가능
- [ ] 금지 물품 카테고리 선택 시 "행정실 전달" 경고 모달
- [ ] 시간당 5건 초과 등록 차단
- [ ] 등록 직후 자기 글 상세 페이지로 리다이렉트

### FR-3. 리스트 / 검색 / 필터

**설명**: 홈과 리스트 화면의 탐색 기능.

**인수 조건**
- [ ] 키워드 검색 (제목·설명, PostgreSQL FTS)
- [ ] 필터: 카테고리, 장소, 기간(7일/30일/전체), 상태
- [ ] 종류 탭: 전체 / 분실물 / 습득물
- [ ] 무한 스크롤 또는 페이지네이션 (20개 단위)
- [ ] 빈 결과 시 안내 + "직접 등록" CTA
- [ ] 반응형 카드: 모바일 1열, 태블릿+ 2열

### FR-4. 상세 화면

**설명**: 게시글 정보 표시 + 액션.

**인수 조건**
- [ ] 사진 갤러리 (스와이프)
- [ ] 공개 정보 전체 표시 (비공개 식별 정보·답변은 비공개)
- [ ] 상태 배지 표시
- [ ] 액션 버튼:
  - 비작성자: "내 물건 같아요" / "이거 제가 주웠어요" / 신고
  - 작성자: 수정 / 삭제 / 상태 변경 (완료 처리)
- [ ] 자기 글이면 검증 시도 기록 확인 가능

### FR-5. 양방향 소유권 검증

**설명**: 진짜 주인만 연락 채널을 받도록 검증.

**인수 조건**
- [ ] 경로 A: 분실자가 습득물 글 → 답변 입력 → 해시 비교
- [ ] 경로 B: 습득자가 분실물 글 → 분실자에게 알림 → 분실자 수동 승인/거절
- [ ] 동일 사용자 동일 글에 24h 내 3회 시도 시 잠금
- [ ] 3회 실패 시 작성자에게 "여러 명 시도 중" 푸시
- [ ] 답변은 bcrypt 해시 저장, 평문 미보관
- [ ] 검증 통과 시 게시글 상태 `active → claiming` 전이
- [ ] 통과한 클레임은 작성자가 최종 승인 시에만 채널 오픈

### FR-6. 연락 (MVP — 메시지 1회)

**설명**: 검증 통과 후 앱 내 메시지 1회 전송.

**인수 조건**
- [ ] 발신자: 검증 통과한 클레이머
- [ ] 메시지 본문 1~500자
- [ ] 작성자만 메시지 열람 가능
- [ ] 작성자가 "수락" 누르면 닉네임 + 메시지 전달
- [ ] 전화번호·외부 링크 패턴 정규식으로 자동 마스킹

### FR-7. 상태 관리

**설명**: 게시글 라이프사이클 추적.

**인수 조건**
- [ ] 상태값: `active / claiming / returned / hidden / deleted`
- [ ] 전이 규칙(아키텍처 7장)을 코드로 강제 (잘못된 전이 차단)
- [ ] 작성자가 `active → returned` 수동 전이 가능
- [ ] 양쪽 "수령 확인" → `claiming → returned`
- [ ] Cron으로 `returned + 30일 → hidden`, `hidden + 60일 → deleted`
- [ ] `returned` 시점에 사진 즉시 삭제

### FR-8. 신고 / 관리

**설명**: 부적절 게시물·사용자 신고와 관리자 처리.

**인수 조건**
- [ ] 신고 사유: 허위 / 부적절 사진 / 개인정보 노출 / 욕설 / 기타
- [ ] 동일 사용자 동일 글 1회만 신고 가능
- [ ] 관리자 콘솔에서 신고 큐 조회 (오래된 순)
- [ ] 관리자 액션: 게시글 숨김 / 삭제 / 사용자 경고·정지
- [ ] 처리 결과를 신고자·작성자에게 통지

### FR-9. 마이페이지

**설명**: 사용자 본인의 활동·설정.

**인수 조건**
- [ ] 내 등록 글 목록 (상태별 탭)
- [ ] 진행 중 매칭 목록 (내가 시도한 클레임 + 받은 클레임)
- [ ] 알림함 (최근 30일)
- [ ] 알림 설정 on/off
- [ ] 데이터 삭제 요청 → 즉시 익명화

### FR-10. PWA 푸시 알림

**설명**: Web Push API 기반 알림.

**인수 조건**
- [ ] PWA 설치 시 푸시 권한 요청
- [ ] 구독 정보 `PUSH_SUB` 테이블 저장
- [ ] 아키텍처 13장 매트릭스의 7가지 이벤트 발송
- [ ] 알림 클릭 시 해당 게시글 상세로 이동
- [ ] 권한 거부 시 앱 내 알림함만 동작

### FR-11. PWA 설치

**설명**: 홈 화면 추가 가능한 PWA.

**인수 조건**
- [ ] `manifest.json` 정의 (이름·아이콘·테마)
- [ ] Service Worker로 앱 셸 캐싱
- [ ] HTTPS 강제
- [ ] Lighthouse PWA 점수 90 이상
- [ ] 오프라인 시 캐시된 홈 화면 노출

### FR-12. 법규 페이지

**설명**: 개인정보·이용약관·유실물법 고지.

**인수 조건**
- [ ] 개인정보 처리방침 페이지 (수집·이용·보관·삭제 명시)
- [ ] 이용약관 페이지
- [ ] 유실물법 안내 페이지 ("7일 이내 시설 관리자 제출" 등)
- [ ] 가입 시 동의 체크박스 (필수/선택 분리)
- [ ] 운영자 연락처

## 7. 비기능 요구사항 (Non-Functional)

| 영역 | 기준 |
|---|---|
| 성능 | 홈 LCP < 2.5s (모바일 3G), API p95 < 500ms |
| 가용성 | 99% 이상 (Vercel + Supabase 가용성 의존) |

| 호환성 | iOS Safari 16.4+, Chrome 110+, Android Chrome |
| 비용 | 무료 티어 내 운영 (학교 1곳, 월 활성 1000명 기준) |
| 데이터 보관 | 본문 90일, 사진 반환 시 즉시, 메시지 90일, 로그 3개월 |

## 11. 릴리즈 기준 (Definition of Done)

MVP가 출시되려면 모두 만족.

- [ ] Playwright E2E 시나리오(등록 → 검증 → 메시지 → 수령 확인) 통과
- [ ] Lighthouse: 성능 80+, PWA 90+, 접근성 90+
- [ ] Sentry 에러율 < 1% (스테이징 1주 관찰)
- [ ] 학교 행정실 검토 통과 + 약관 페이지 확정
